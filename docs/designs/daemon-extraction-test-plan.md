## Affected Pages/Routes

- `GET /api/health` — Verify daemon responds with version, uptime, feature flags
- `GET /api/repos` — List repos (empty + populated cases)
- `POST /api/repos` — Add repo (valid path + invalid path)
- `GET /api/repos/:id/worktrees` — List worktrees (repo exists + 404)
- `POST /api/repos/:id/worktrees` — Create worktree (success + branch conflict 409)
- `WS /api/ws` — WebSocket connects, receives events when state changes
- `POST /api/runs` — Launch run (single + parallel)
- `GET /api/runs` — List all active runs
- `GET /api/runs/:id` — Get specific run status
- `POST /api/runs/:id/cancel` — Cancel specific run
- `POST /api/orchestrate` — Launch cross-repo orchestration (valid + invalid repos)
- `GET /api/orchestrate/:id` — Orchestration status (in-progress + completed)
- `POST /api/orchestrate/:id/cancel` — Cancel orchestration
- `GET /api/knowledge/search` — Knowledge query across repos

## Key Interactions to Verify

- Daemon starts and binds to localhost only on configured port
- EventEmitter trait correctly routes events to WebSocket clients
- SidecarHandle trait spawns/manages Node.js processes per run
- Parallel runs: two runs active simultaneously, each with independent lifecycle
- Permission requests routed correctly with compound key (session_id, tool_use_id)
- Permission dialog includes run context (task description, PR info)
- Orchestration creates worktrees across multiple repos and launches parallel runs
- WebSocket events scoped by run_id

## Edge Cases

- Launch run when daemon has no repos configured (400)
- Cancel a run that already completed (idempotent or 409)
- WebSocket client disconnects mid-run (server continues, no crash)
- Orchestrate with a mix of valid and invalid repo paths (partial failure handling)
- Two permission requests from different parallel runs arrive simultaneously
- Daemon port already in use on startup (clear error message)
- Sidecar process crashes mid-run (run marked failed, cleanup)

## Critical Paths

- External script: `curl /api/health` → `curl POST /api/repos` → `curl POST /api/repos/:id/worktrees` → verify via `curl GET /api/repos/:id/worktrees` (smoke test for extraction)
- Full orchestration: `POST /api/orchestrate` → WS subscribe → see N runs progress → all complete → `GET /api/orchestrate/:id` shows success
- Parallel permission: two runs active → both request permission → user approves one, denies other → correct runs get correct responses
