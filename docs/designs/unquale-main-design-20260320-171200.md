# Design: Multi-Repo Orchestration Platform — Daemon Extraction + External API

Generated by /office-hours on 2026-03-20
Branch: main
Repo: Branchdeck
Status: APPROVED
Mode: Builder

## Problem Statement

Branchdeck is a capable multi-worktree terminal manager with Claude integration and a local knowledge layer. But its current architecture is Tauri-locked — all operations flow through IPC, one run at a time, no external access. To become an orchestration platform that AI/agent builders want to build on top of, Branchdeck needs to extract its service layer into a standalone daemon with an HTTP + WebSocket API, support parallel agent runs across repos, and expose a surface that external orchestrators can drive programmatically.

## What Makes This Cool

The "whoa" moment: an external script (or another Claude Code session, or a higher-level orchestrator like OpenClaw) sends a single API call — "update this shared library across 4 downstream repos" — and Branchdeck creates 4 worktrees, launches 4 parallel Claude Code sessions with knowledge context, monitors all of them in a live dashboard, captures artifacts, and reports results. All programmatically. All locally. No cloud. No vendor lock-in.

For AI/agent builders, this is the missing piece: a local execution environment purpose-built for multi-repo agent orchestration, with a real API surface, persistent knowledge, and crash-proof state.

## Constraints

- **Rust expertise:** User is new to Rust — architecture should minimize complex Rust patterns (no custom async runtimes, minimal unsafe, lean on existing libraries)
- **Package manager:** Bun only (never npm/yarn/pnpm)
- **No business logic in Tauri commands:** Already enforced — commands validate args, call services, return results
- **Services are daemon-ready:** No Tauri imports in `services/`. This is the foundation for extraction
- **File-based persistence:** task.md + run.json + config files. No database
- **Single active run currently:** RunManager supports one active run. PR Shepherd spec adds sequential queue. Parallel is new
- **Knowledge is feature-gated:** Behind `knowledge` flag. Not a blocker but needs consideration during extraction
- **Linux-first:** Tauri + Axum both work well on Linux. No cross-platform surprises expected

## Premises

1. **The external API is the unlock, not the UI.** For AI/agent builders, programmatic access matters more than dashboard polish. The dashboard is monitoring, not the primary interface.
2. **Sequential before parallel.** Ship sequential batch first, then add parallel. Validate orchestration UX before scaling it.
3. **The knowledge layer differentiates but doesn't block.** Knowledge works. It needs depth later, but orchestration can demo without perfect knowledge flow.
4. **Daemon extraction is the first domino.** Can't have an external API without extracting from Tauri. Services are ready. This unblocks everything.
5. **Do nothing = stays a tool, not a platform.** The orchestration layer is what makes the "whoa" for AI/agent builders.

## Approaches Considered

### Approach A: Daemon-First (Chosen)

Extract Rust services into a standalone Axum HTTP + WebSocket daemon. Tauri becomes a thin client. External orchestrators connect to the same API. Add parallel RunManager on top.

- **Effort:** L (human: ~3-4 weeks / CC: ~6-8 hours)
- **Risk:** Medium — big move, but services are already daemon-ready
- **Pros:** Clean architecture, enables hosted path, API is native, unblocks everything
- **Cons:** Longest path to first demo, touches every layer
- **Reuses:** Existing service layer, sidecar bridge, task.md persistence, knowledge MCP server
- **Completeness:** 9/10

### Approach B: HTTP Sidecar API

Skip daemon extraction. Add Axum HTTP server as a second sidecar alongside Node.js agent bridge. Proxy to Tauri's internal commands.

- **Effort:** M (human: ~1-2 weeks / CC: ~3-4 hours)
- **Risk:** Low — additive, no restructuring
- **Pros:** Fastest path to external API
- **Cons:** Architectural debt, doesn't enable hosted path, limited by Tauri command surface
- **Completeness:** 6/10

### Approach C: Agent-Native (MCP + Skill File)

No HTTP API. Orchestration via Claude Code skill file + expanded MCP server. External "orchestrators" are Claude Code sessions calling Branchdeck's MCP tools.

- **Effort:** M (human: ~2 weeks / CC: ~4-5 hours)
- **Risk:** Medium — novel, Claude-locked
- **Pros:** Audience-native, knowledge integration natural, genuinely novel demo
- **Cons:** Claude ecosystem lock-in, limited vs. HTTP
- **Completeness:** 7/10

## Recommended Approach

**Approach A: Daemon-First.** The services are already daemon-ready (no Tauri imports). The effort delta between A and B is ~4 hours with CC, but A eliminates the architectural debt and unblocks the hosted path, external orchestrator API, and parallel execution. You'd pay back B's debt eventually; better to do it right now while the cost is low.

## Architectural Decisions (from eng review 2026-03-20)

### D1: EventEmitter trait for Tauri decoupling

6 service files call `app_handle.emit()` directly. Extract an `EventEmitter` trait in branchdeck-core:

```rust
pub trait EventEmitter: Send + Sync {
    fn emit_status_changed(&self, info: &RunInfo);
    fn emit_permission_request(&self, perm: &PendingPermission);
    fn emit_event(&self, name: &str, payload: &impl serde::Serialize);
}
```

Two implementations:
- `TauriEmitter` (in src-tauri) — wraps `AppHandle::emit()`
- `DaemonEmitter` (in branchdeck-daemon) — publishes to EventBus → WebSocket

### D2: Tauri calls services in-process (not via daemon HTTP)

Tauri imports branchdeck-core directly. Daemon imports branchdeck-core separately. Desktop app works standalone without daemon. Daemon is optional, for external API consumers.

Future: when hosted version ships, Tauri can optionally connect to remote daemon. Design for in-process now, keep HTTP option open.

Daemon should be containerizable (Docker) for the VPS hosted path.

### D3: SidecarHandle trait for run isolation

```rust
pub trait SidecarHandle: Send + Sync {
    async fn spawn(task_path: &str, worktree_path: &str) -> Result<Self, AppError>;
    async fn send(&self, msg: SidecarRequest) -> Result<(), AppError>;
    async fn cancel(&self) -> Result<(), AppError>;
    fn on_event(&self, callback: impl Fn(SidecarResponse) + Send + 'static);
}
```

First implementation: `ProcessSidecarHandle` — one Node.js child process per run. Zero JS changes to agent-bridge.js. ~50MB memory per run, fine for 2-4 parallel runs.

Door open for `MultiplexedSidecarHandle` later (fix JS globals, route by session ID). Zero RunManager changes when swapped.

### D4: Permission routing with compound key

Permissions keyed by `(session_id, tool_use_id)` instead of `tool_use_id` alone. Prevents collision across concurrent sessions.

Permission event payload must include full run context: session_id → run → task → PR number/description. User sees "PR #42 shepherd wants to run git push", not just "something wants to run git push".

### D5: Staged API rollout

Step 1 ships minimal API to prove extraction:
```
GET  /api/health                    (version, uptime, feature flags)
GET  /api/repos
POST /api/repos
GET  /api/repos/:id/worktrees
POST /api/repos/:id/worktrees
WS   /api/ws
```

Step 2 adds run endpoints. Step 3 adds orchestration + knowledge.

### D6: Cargo workspace structure

```
Branchdeck/
├── Cargo.toml              ← workspace root
│   [workspace]
│   members = ["src-tauri", "crates/*"]
│   resolver = "2"
├── crates/
│   ├── branchdeck-core/    ← models, services, error, EventEmitter, SidecarHandle
│   └── branchdeck-daemon/  ← Axum HTTP+WS server, DaemonEmitter, DaemonError
├── src-tauri/              ← stays in place (Tauri CLI requires this path)
├── sidecar/                ← unchanged
└── src/                    ← frontend, unchanged
```

Feature flag: `knowledge` feature lives on branchdeck-core. Consumers opt in:
```toml
branchdeck-core = { path = "../crates/branchdeck-core", features = ["knowledge"] }
```

CI: `cargo clippy --workspace` / `cargo fmt --all --check` / `cargo test --workspace` from repo root.

Test structure: branchdeck-core/tests/ (service unit tests, moved from src-tauri/tests/), branchdeck-daemon/tests/ (HTTP route tests), src-tauri/tests/ (Tauri-specific only).

### D7: DaemonError wrapper for HTTP status codes

AppError stays transport-agnostic in branchdeck-core. branchdeck-daemon defines `DaemonError(AppError)` with `impl IntoResponse` that maps variants to HTTP status codes.

### D8: tokio::sync::Mutex for RunManager

RunManager uses `tokio::sync::Mutex` (yields instead of blocking tokio workers). Required because handle_response is async and lock guards span .await points.

All other Mutex<T> uses (TerminalService, config) keep `std::sync::Mutex` — sync code with microsecond hold times.

### D9: Orchestration two-phase failure model

**Validation phase (before any run starts):** Fail fast. Repo doesn't exist, worktree can't be created, PR not found → reject entire orchestration. 400 with specifics.

**Execution phase (after runs start):** Independent per-run outcomes. Run A succeeds, run B fails, run D still going. No rollback. Orchestration status reports per-run results.

Rule: validate everything upfront, then let execution be independent.

### D10: Atomic PR for workspace restructure

The workspace restructure PR must include in one atomic commit:
- Root Cargo.toml (workspace definition)
- crates/branchdeck-core/ with extracted code
- Updated src-tauri/Cargo.toml
- Updated .github/workflows/ci.yml
- Updated CLAUDE.md
- Updated README.md (if development section references cd src-tauri)

### D11: Worktree mutual exclusion on run launch (from CEO review)

Reject `POST /api/runs` if an active run already targets the same worktree path. Return 409 Conflict. Two Claude Code sessions writing to the same worktree simultaneously is a race condition — file corruption, git conflicts, corrupted state.

Check `active_runs` HashMap for matching `worktree_path` at launch time. Same check applies to `POST /api/orchestrate` during the validation phase.

### D12: Request ID middleware (from CEO review)

Every HTTP response includes `X-Request-Id: uuid`. Daemon logs include the same ID. Axum middleware using `tower_http`. Enables log correlation when debugging API issues.

### D13: Startup banner + request tracing (from CEO review)

Daemon prints structured startup info:
```
Branchdeck Daemon v0.2.0
Listening on http://127.0.0.1:13371
Features: knowledge=true, parallel_runs=true
Repos loaded: 3
Ready.
```

Plus `tower_http::trace::TraceLayer` for every request (method, path, status, duration, request_id).

### D14: Smoke test script (from CEO review)

`scripts/smoke-test.sh` ships with Step 1. Starts daemon, curls every endpoint, validates responses, stops daemon. Foundation for CI daemon integration tests.

### D15: README API Quick Start (from CEO review)

5-line curl example in README showing: start daemon, add repo, list worktrees. Viral growth vector for AI/agent builders.

## Implementation Staircase

### Step 1: Daemon Extraction (~2-3 hours CC)

**Goal:** Standalone Axum binary serving minimal API to prove extraction works.

**Crate extraction:**
- Create Cargo workspace (D6) with branchdeck-core and branchdeck-daemon crates
- Move to branchdeck-core (zero changes needed): all models (6 files), error.rs, EventBus, config.rs, git.rs, github.rs, task.rs, hook_config.rs, hook_receiver.rs, run_state.rs, activity_store.rs, agent_scanner.rs, knowledge_*.rs
- Define EventEmitter trait (D1) and SidecarHandle trait (D3) in branchdeck-core
- Refactor 6 Tauri-coupled services to use EventEmitter: run_manager.rs, run_effects.rs (execute_effects only), run_responses.rs, run_stale.rs, task_watcher.rs, knowledge_ingestion.rs (tauri::async_runtime → tokio::spawn)
- src-tauri adds TauriEmitter implementing EventEmitter
- Move existing tests from src-tauri/tests/ to crates/branchdeck-core/tests/

**Daemon server:**
- branchdeck-daemon: Axum HTTP+WS on configurable port (default: 13371), localhost-only binding
- DaemonEmitter implementing EventEmitter (publishes to EventBus → WebSocket)
- DaemonError wrapping AppError with IntoResponse (D7)
- Step 1 API endpoints only (D5): health, repos, worktrees, WebSocket
- HTTP route tests with mock traits + tempdir git repos

**Atomic PR (D10):** workspace + crate extraction + CI updates + CLAUDE.md + README.md

**Smoke test:** `curl localhost:13371/api/health` → version + feature flags. `curl POST /api/repos` → add repo. `curl GET /api/repos/:id/worktrees` → list worktrees.

### Step 2: Parallel RunManager (~1-2 hours CC)

**Goal:** Support N concurrent agent runs, each in its own worktree.

- Replace `active_run: Option<RunInfo>` with `active_runs: HashMap<RunId, RunInfo>`
- Implement ProcessSidecarHandle (D3): one Node.js process per run, zero JS changes
- Switch RunManager to `tokio::sync::Mutex` (D8)
- Permission routing with compound key `(session_id, tool_use_id)` (D4)
- Permission events include full run context chain (session → run → task → PR)
- Run lifecycle (launch → running → completed/failed/cancelled) per-run, not global
- Add run API endpoints: `POST /api/runs`, `GET /api/runs/:id`, `POST /api/runs/:id/cancel`, `GET /api/runs`
- Events scoped by run ID on WebSocket
- Sequential queue still available as a mode (batch with concurrency=1)

### Step 3: External Orchestration API (~1-2 hours CC)

**Goal:** Complete API surface for external systems to drive Branchdeck.

- `POST /api/orchestrate` — high-level: "apply this change across these repos"
  - Two-phase failure model (D9): validate all repos/worktrees upfront (400 on any invalid), then launch independent runs
  - Creates worktrees, launches parallel runs, returns orchestration ID
  - WebSocket streams progress per-run
- `GET /api/orchestrate/:id` — status of all runs in an orchestration (per-run results)
- `POST /api/orchestrate/:id/cancel` — cancel all runs in orchestration
- `GET /api/knowledge/search` — query knowledge across repos (expose existing MCP tools via HTTP)

### Step 4: Knowledge Flow Across Runs (~1 hour CC)

**Goal:** What one agent learns surfaces in another agent's context.

- Post-run artifact capture already works → ensure captured knowledge is repo-scoped AND global-scoped
- Pre-run knowledge injection: query knowledge for the specific repo + global context, inject into agent's system prompt / task.md
- Cross-repo knowledge: when orchestrating N runs, each run gets knowledge from its own repo + knowledge from sibling runs that completed first
- Simple implementation: after each run completes, capture artifacts → next run in the batch gets updated knowledge

### Step 5: Dashboard Monitoring UI (~1-2 hours CC)

**Goal:** Visual monitoring surface for orchestrations in progress.

**Prerequisite:** Run `/design-consultation` to create DESIGN.md before implementing this step.

#### Layout & Navigation (from design review 2026-03-21)

**Dedicated orchestration view:** Full app takeover — replaces the Shell layout entirely when viewing orchestrations. Not a panel within the existing 3-panel layout.

**Navigation:** Tab/mode switcher in the TopBar: `[Workspace] [Orchestrations (N)]`. User clicks to switch between normal workspace and orchestration view. Badge shows count of active + recent orchestrations.

**Orchestration concurrency:** One active orchestration at a time. The Orchestrations tab shows:
- **Active section** (if running): orchestration name, progress summary, [View Dashboard] button
- **History section:** completed orchestrations with status, run count, total time

#### Information Hierarchy

Three levels of detail, each progressively deeper:

1. **Orchestration header** (primary) — name, overall progress ("3/4 complete"), total elapsed time
2. **Run grid** (secondary) — per-run cards in a CSS grid (`auto-fill`, `minmax(300px, 1fr)`)
3. **Expanded terminal** (tertiary) — click any run card to expand its full xterm.js terminal below the grid

#### Run Card Design

Each run card shows:
- **Header:** repo name + status badge (RUNNING/OK/FAIL/CANCELLED)
- **Running state:** elapsed time + live indicator dot + last 3-5 lines of terminal output scrolling in real-time (mini terminal preview)
- **Completed state:** elapsed time + artifact count + PR link (if created)
- **Failed state:** red badge + truncated last error line inline (e.g. "Error: merge conflict in lib/shared.ts") + elapsed time
- Click any card to expand its full terminal below the grid

#### Interaction States

| FEATURE | LOADING | EMPTY | ERROR | SUCCESS | PARTIAL |
|---------|---------|-------|-------|---------|---------|
| Orchestration list | Skeleton cards | Instructional empty state: curl example showing how to trigger orchestration via API, with [Open API docs] and [Copy curl] buttons | "Daemon not running" with start instructions | List of orchestration cards with status | N/A |
| Run grid | Pulsing placeholder cards matching expected run count | N/A (always has runs once orchestration starts) | Per-run error badge (red) with last error line inline | Green checkmark badge per run | Mix of completed/running/failed badges |
| Individual run card | Spinner + "Launching..." | N/A | Red badge + last error line from terminal | Green badge + artifact count + elapsed time | Running badge + elapsed time + live terminal preview |
| Expanded terminal | "Connecting to terminal..." | Cursor blinking, awaiting output | Terminal shows error output; banner "Run failed" | Terminal shows final output; banner "Run completed" | Live streaming output |
| Status bar | "Starting orchestration..." | N/A | "Orchestration failed: 2/4 runs errored" | "All 4 runs completed successfully" | "3/4 completed, 1 in progress" |

#### Completion Summary Screen

When all runs finish, transition from grid view to a summary screen:
- Orchestration name + total time + run count
- Per-repo results: status icon + repo name + outcome (e.g. "PR #42 created" or "merge conflict")
- Aggregate stats: PRs created, artifacts captured, failures
- Action buttons: [View PRs] [Retry Failed]

#### WebSocket Disconnect Handling

Banner at top: "Connection lost -- reconnecting..." with auto-retry every 3s. Run cards show last-known state with "?" for unknown runs. On reconnect, refresh full state from REST API.

#### Keyboard Navigation

- Tab: navigate between run cards in grid
- Enter: expand selected run's terminal
- Esc: collapse expanded terminal / go back
- No full ARIA spec for now (desktop dev tool)

#### Responsive Behavior

CSS grid with `auto-fill`, `minmax(300px, 1fr)` — adapts to window width automatically. No explicit breakpoints needed for a desktop app.

#### Visual Identity

Terminal-first aesthetic: dark theme, monospace density, minimal chrome. Think `htop`/`tmux` command center, not Grafana dashboard. Reuse existing xterm.js terminal component and status color palette (red-400/yellow-400/green-400). Avoid generic card-grid SaaS patterns.
