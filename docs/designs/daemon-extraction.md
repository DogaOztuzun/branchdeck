## Problem Statement

Branchdeck is a capable multi-worktree terminal manager with Claude integration and a local knowledge layer. But its current architecture is Tauri-locked — all operations flow through IPC, one run at a time, no external access. To become an orchestration platform that AI/agent builders want to build on top of, Branchdeck needs to extract its service layer into a standalone daemon with an HTTP + WebSocket API, support parallel agent runs across repos, and expose a surface that external orchestrators can drive programmatically.

## What Makes This Cool

The "whoa" moment: an external script (or another Claude Code session, or a higher-level orchestrator like OpenClaw) sends a single API call — "update this shared library across 4 downstream repos" — and Branchdeck creates 4 worktrees, launches 4 parallel Claude Code sessions with knowledge context, monitors all of them in a live dashboard, captures artifacts, and reports results. All programmatically. All locally. No cloud. No vendor lock-in.

For AI/agent builders, this is the missing piece: a local execution environment purpose-built for multi-repo agent orchestration, with a real API surface, persistent knowledge, and crash-proof state.

## Constraints

- **Rust expertise:** User is new to Rust — architecture should minimize complex Rust patterns (no custom async runtimes, minimal unsafe, lean on existing libraries)
- **Package manager:** Bun only (never npm/yarn/pnpm)
- **No business logic in Tauri commands:** Already enforced — commands validate args, call services, return results
- **Services are daemon-ready:** No Tauri imports in `services/`. This is the foundation for extraction
- **File-based persistence:** task.md + run.json + config files. No database
- **Single active run currently:** RunManager supports one active run. PR Shepherd spec adds sequential queue. Parallel is new
- **Knowledge is feature-gated:** Behind `knowledge` flag. Not a blocker but needs consideration during extraction
- **Linux-first:** Tauri + Axum both work well on Linux. No cross-platform surprises expected

## Architectural Decisions (D1-D15)

### D1: EventEmitter trait for Tauri decoupling
6 service files call `app_handle.emit()` directly. Extract `EventEmitter` trait in branchdeck-core. Two impls: `TauriEmitter` (src-tauri), `DaemonEmitter` (branchdeck-daemon → EventBus → WebSocket).

### D2: Tauri calls services in-process
Both Tauri and daemon import branchdeck-core directly. Desktop app works standalone. Daemon is optional for external API consumers. Containerizable (Docker) for VPS hosted path.

### D3: SidecarHandle trait for run isolation
First impl: `ProcessSidecarHandle` — one Node.js child per run. Zero JS changes. ~50MB/run. Door open for `MultiplexedSidecarHandle` later.

### D4: Permission routing with compound key
`(session_id, tool_use_id)` prevents collision. Permission events include full run context chain (session → run → task → PR).

### D5: Staged API rollout
Step 1: health + repos + worktrees + WS. Step 2: runs. Step 3: orchestrate + knowledge.

### D6: Cargo workspace structure
```
Branchdeck/
├── Cargo.toml              ← workspace root [members = ["src-tauri", "crates/*"]]
├── crates/
│   ├── branchdeck-core/    ← models, services, error, traits
│   └── branchdeck-daemon/  ← Axum HTTP+WS, DaemonEmitter, DaemonError
├── src-tauri/              ← stays in place (Tauri CLI requirement)
├── sidecar/                ← unchanged
└── src/                    ← frontend, unchanged
```

### D7: DaemonError wrapper for HTTP status codes
AppError stays transport-agnostic. DaemonError(AppError) in daemon crate with `impl IntoResponse`.

### D8: tokio::sync::Mutex for RunManager
Async code holding locks across .await points. std::sync::Mutex for everything else.

### D9: Orchestration two-phase failure model
Validate-all-or-reject upfront (400). Independent per-run outcomes during execution.

### D10: Atomic PR for workspace restructure
One PR: workspace + crate extraction + CI updates + CLAUDE.md + README.md.

### D11: Worktree mutual exclusion on run launch
Reject duplicate worktree runs with 409 Conflict. Check active_runs HashMap at launch time.

### D12: Request ID middleware
`X-Request-Id: uuid` in every HTTP response + daemon logs. Axum middleware via tower_http.

### D13: Startup banner + request tracing
Structured startup log (version, port, features, repo count) + TraceLayer for all requests.

### D14: Smoke test script
`scripts/smoke-test.sh` ships with Step 1. Starts daemon, curls endpoints, validates, stops.

### D15: README API Quick Start
5-line curl example in README. Viral growth vector for AI/agent builders.

## Implementation Staircase

### Step 1: Daemon Extraction (~2-3 hours CC)
- Cargo workspace (D6) + branchdeck-core + branchdeck-daemon crates
- Move ~14 clean service files to core. Refactor 6 coupled files via EventEmitter trait (D1)
- Axum server: health, repos, worktrees, WebSocket (D5)
- DaemonEmitter, DaemonError, request ID middleware (D7, D12, D13)
- Smoke test script (D14), README API Quick Start (D15)
- Atomic PR (D10)

### Step 2: Parallel RunManager (~1-2 hours CC)
- HashMap<RunId, RunInfo>, ProcessSidecarHandle (D3), tokio::sync::Mutex (D8)
- Compound permission keys (D4), worktree mutual exclusion (D11)
- Run API endpoints, WS events scoped by run_id

### Step 3: External Orchestration API (~1-2 hours CC)
- POST /api/orchestrate with two-phase failure model (D9)
- GET status, POST cancel, GET knowledge/search
- Per-run results in orchestration status

## API Surface (Staged)

```
# Step 1
GET  /api/health          → { version, uptime, features }
GET  /api/repos           → Vec<Repo>
POST /api/repos           → Repo
GET  /api/repos/:id/worktrees → Vec<Worktree>
POST /api/repos/:id/worktrees → Worktree
WS   /api/ws              → Stream<Event>

# Step 2
POST /api/runs            → RunInfo
GET  /api/runs/:id        → RunInfo
POST /api/runs/:id/cancel → ()
GET  /api/runs            → Vec<RunInfo>

# Step 3
POST /api/orchestrate     → OrchestrationInfo
GET  /api/orchestrate/:id → OrchestrationInfo
POST /api/orchestrate/:id/cancel → ()
GET  /api/knowledge/search → Vec<KnowledgeEntry>
```
